# Vault rules — global

This file defines the global rules every LLM session that touches this vault must follow. Per-app rules live in the `CLAUDE.md` inside each application's working directory (dropped there by `setup-server.sh`).

## Purpose

This vault is the source of truth for the architecture of an ecosystem of N applications running across M servers, with bilateral integrations between them documented as first-class pages. The pattern is: **LLM Wiki** (Karpathy) extended for **multi-server** use (VaultMesh) — `raw/` for source material, `wiki/` for distilled knowledge.

## Never write to the vault

- Passwords, tokens, API keys, connection strings
- Contents of `.env`, `config/app.php`, `config/database.php`, `.htpasswd`, etc.
- Real customer data (names, IDs, addresses, phone numbers, emails)
- Production SQL dumps
- Anything that would be dangerous if the repo became public by accident

If you need to illustrate an example, use fictional data and label it as such.

## Structure

```
wiki/
├── index.md              regenerated by lint, do NOT edit by hand
├── log.md                append-only session journal
├── apps/{app-name}/      owned by that application
│   ├── README.md         app overview
│   ├── modules/          one page per module/feature
│   ├── debugging/        resolved bugs (format: YYYY-MM-slug.md)
│   ├── runbooks/         "how to do X when Y happens"
│   └── proposed-adr/     ADR proposals (drafts)
├── integrations/         contracts between apps
│   └── NN-producer--consumer--subject.md
├── flows/                end-to-end journeys (cross-app)
├── infrastructure/       per-server documentation
├── shared/               cross-cutting knowledge (shared schemas, auth, deployment)
└── decisions/            global architectural ADRs (promoted from proposed-adr)
```

## Ownership and scope

- Each app **owns** `wiki/apps/{itself}/**` and writes there freely.
- `wiki/integrations/` — the **producer** owns the contract page. The **consumer** writes ONLY in the "Consumer notes" section.
- `wiki/flows/` — the flow is owned by the app that initiates the journey; other participants fill in their sections.
- `wiki/decisions/` — DO NOT write here directly. Proposals go in `wiki/apps/{app}/proposed-adr/`; a human promotes them after discussion.
- `wiki/index.md` — DO NOT edit by hand (regenerated by lint).

The exact write permissions per app are encoded in each app's per-directory `CLAUDE.md`.

## Required frontmatter

Every page in `wiki/` starts with:

```yaml
---
title: [short, descriptive title]
type: app | integration | flow | adr | runbook | debugging | module
status: draft | active | deprecated | legacy
owners: [list of apps, e.g. [inventory]]
updated: YYYY-MM-DD
---
```

For integrations, also include:

```yaml
producer: [app]
consumer: [app]
transport: rest | soap | shared-db | cron-file | cron-fetch | queue
```

## Wikilinks

- Internal references: `[[apps/inventory/modules/stock-adjustments]]` — relative path from `wiki/`, no `.md` extension.
- DO NOT use absolute URLs for internal references.
- DO NOT use leading-slash paths.
- Contract references: `[[integrations/01-inventory--pos--product-data]]`.

## Content conventions

### `apps/{app}/README.md`
Overview of the app: role, stack, main modules (with links), integrations where it's a producer, integrations where it's a consumer, available runbooks. See `_templates/app.md`.

### Pages under `modules/`
Every major module/feature gets its own page: what it does, what tables it touches, what endpoints it exposes, what integrations it uses.

### Pages under `debugging/`
Format: `YYYY-MM-slug.md`. Bugs that took longer than 30 minutes or are non-trivial. Structure: symptom → root cause → fix → prevention. A goldmine six months later.

### Pages under `integrations/`
Filename: `NN-producer--consumer--subject.md` where NN is two digits (01–99). See `_templates/integration.md`. The "Breaking changes log" section is updated by the producer; the consumer adds edge cases under "Consumer notes".

### Pages under `flows/`
Journeys that traverse 2+ apps. Structured chronologically, with sections per participating app and links to relevant integration contracts.

### ADRs (`decisions/`)
Format: `YYYY-MM-slug.md`. Structure: Context / Decision / Alternatives / Consequences. See `_templates/adr.md`.

## The log

At the end of every session that touched code or the vault, append an entry to `log.md`:

```
## YYYY-MM-DD HH:MM — {app} on Server {N}
- modules touched: [list]
- integrations verified/updated: [list]
- notable decisions or debugging added: [list or -]
```

This is the single most important convention in the whole vault. It's what makes the wiki feel alive and what gives every server visibility into what the others are doing.

## Production code

LLM sessions read production code to understand it, but **DO NOT** copy large code blocks into the vault. Document **behavior** and **decisions**, not implementation verbatim. Implementation lives in the app's own repo. The wiki answers "what" and "why", not "how exactly" — that answer comes from the code.

Reasonable exceptions: small fragments (≤ 10 lines) to illustrate a pattern or convention, with a clear reference to the source file.

## Vault sync between servers (git workflow)

The vault lives as a git repo cloned on every server, with a central remote. Two shell commands are available in the server shell:

- `vs` — vault-sync, runs `git pull --rebase --autostash` in the vault path
- `vp "msg"` — vault-push, does status + confirmation + commit + push

### When to run `vs` autonomously

At the start of a session, before reading or writing any vault content, run `vs` so you're working with the current version. Sync is read-only and safe — no user confirmation needed. Examples of when:

- The user asks something that involves reading the vault ("what do we know about X?")
- The user asks you to modify a vault page
- You're starting a documentation session

If `vs` returns a conflict (extremely rare), STOP, report which file is in conflict to the user, and wait for instructions — DO NOT resolve it on your own.

### When to use `vp` (REQUIRES confirmation)

`vp` writes to the remote. NEVER run it without explicit user confirmation in chat. Correct pattern:

1. After modifying vault content, summarize for the user what you wrote.
2. Propose the exact command, e.g. `vp "inventory: document daily reconciliation cron"`.
3. Wait for the user to confirm explicitly ("yes", "go", "ok") OR for them to run it manually.
4. Only after confirmation, run the command.

`vp` is FORBIDDEN for:
- Automatic "tidy-up" commits at session end without approval
- Auto-retry if it failed once (the user must see the error)
- Multiple messages in a single batch with no review pause

### Commit message convention

Format: `<app>: <short imperative action>`, first line under 60 chars.

Good:
- `inventory: document StockAdjustments module`
- `pos: add debugging page for 2026-04 stock-drift incident`
- `webstore: clarify shared-DB table ownership`

Avoid:
- `update docs` (too vague)
- `various changes` (hides scope)
- Mixed-language messages

If a change touches multiple apps (rare), use `cross:` as the prefix.

### When `vp` rejects the push

If push fails with "fetch first" / "rejected", another server pushed in the meantime. Procedure:

1. Run `vs` (it will rebase automatically)
2. Check `git log --oneline -5` to see what arrived
3. If everything looks OK, re-propose `vp` with the same message for confirmation
4. If rebase produced a conflict, STOP and report to the user

### What we do NOT do with git in the vault

- No `git reset --hard` without confirmation — loses local work
- Never `git push --force` (the vault doesn't tolerate history rewrites)
- No `git commit --amend` after a push
- Don't touch `.git/hooks/` — the security hook is critical

## When in doubt

1. Check the `CLAUDE.md` in your application's working directory for scoped rules.
2. If a change touches another app, **don't make it** — write a proposed-ADR in `wiki/apps/{your-app}/proposed-adr/` and let a human coordinate.
3. The breaking-changes log on a contract is updated only by the producer.

## End-of-session checklist

- [ ] Pages in `wiki/apps/{your-app}/` you touched have `updated: ...` refreshed in frontmatter
- [ ] Bugs that took > 30 min → page in `debugging/`
- [ ] Architectural decisions → proposal in `proposed-adr/`
- [ ] Modified contracts → entry in the contract's "Breaking changes log" (producer only)
- [ ] New entry in `log.md`
